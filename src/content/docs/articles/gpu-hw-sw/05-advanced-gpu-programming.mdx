## Lesson 1: 고급 GPU 프로그래밍

### 강의 학습 목표

- GPU 최적화 기법을 설명한다
- 고급 기법 자료를 이해하기 위한 기반을 갖춘다

### GPU에 적합한 애플리케이션

GPU는 다음 특성을 가진 작업에 적합하다:

- 대규모 스레드를 활용할 수 있는 **대량 병렬 데이터**를 포함하는 작업
- 호스트-디바이스 간 통신 비용이 **상대적으로 낮은** 애플리케이션
- 전역 메모리 접근이 **코얼레싱(coalesced)** 되는 애플리케이션

### 프로파일링

프로파일링은 애플리케이션의 핫스팟을 식별하여 성능 병목을 찾아내는 도구다. 주요 성능 지표:

- 보고된 처리량(throughput)
- 분기 발산(divergent branch) 횟수
- 메모리 발산 횟수 (코얼레싱/비코얼레싱)
- 점유율(occupancy)
- 메모리 대역폭 활용률

GPU 벤더가 제공하는 프로파일러를 사용하여 커널 함수를 분석·최적화한다. GPU 변환 전에 CPU 핫스팟을 식별하는 데에도 프로파일링이 유용하다.

### 최적화 기법 개요

GPU 실행 시간은 **데이터 전송 시간**, **메모리 접근 시간**, **연산 시간** 세 요소로 분해된다. 따라서 최적화도 세 방향으로 접근한다:

1. 데이터 전송 시간 감소
2. 메모리 접근 패턴 최적화
3. 연산 오버헤드 감소

가능하면 제공되는 **라이브러리를 활용**하는 것이 좋다.

### 데이터 전송 최적화

호스트-디바이스 간 데이터 전송 오버헤드 최적화가 중요하다. 전송 오버헤드는 호스트-디바이스 메모리 연결 패브릭(일반적으로 PCI-E)에서 발생하며, 오버헤드가 크다.

**최적화 방법:**

**1. 고속 전송 방법 사용 — 피닝된 메모리(Pinned Memory)**

피닝된 메모리를 사용하면 CPU 측 오버헤드를 줄여 데이터 복사 시간이 감소한다.

**2. 연산과 데이터 전송의 중첩**

`cudaMemcpyAsync` API로 데이터 복사와 커널 실행을 중첩할 수 있다.

**3. 데이터 전송과 연산의 파이프라이닝**

CUDA의 **스트림(stream)** 기능으로 동시 복사와 실행을 구현한다. 커널과 복사를 여러 세션으로 분할해야 하는 경우가 많다.

**4. 직접 호스트 메모리 접근**

- **제로 카피(Zero Copy)**: CPU-GPU 통합 메모리에서 CPU 데이터에 직접 접근한다.
- **통합 가상 주소 지정(UVA, Unified Virtual Addressing)**: 드라이버와 런타임이 물리적으로 분리된 메모리 공간을 숨기고, CPU와 GPU가 같은 메모리 공간을 공유하는 것처럼 인터페이스를 제공한다. 이상적으로는 데이터 복사 비용이 없어야 하지만, 구현상 여전히 데이터 복사가 필요하므로 오버헤드가 존재한다.

모든 방법이 모든 애플리케이션에 적용 가능한 것은 아니다. 시스템과 대상 애플리케이션에 따라 다양한 방법을 시도해야 한다.

### 메모리 접근 패턴 최적화

전역 메모리 접근 오버헤드를 줄이는 것이 가장 중요한 최적화 기회 중 하나다.

- 가능한 한 **캐시를 최대한 활용**한다
- 모든 전역 메모리 접근이 **코얼레싱되고 정렬(aligned)** 되도록 보장한다
- 데이터 메모리 **트랜잭션 횟수를 줄이는 것**이 성능 최적화의 핵심이다
- 평균 메모리 대역폭 소비량을 확인하여 메모리 피크 대역폭을 충분히 활용하고 있는지 점검한다
- **공유 메모리 뱅크 충돌을 줄인다**

### 연산 오버헤드 감소

- **명령어 수준 최적화**: 고비용 명령어를 저비용 명령어로 대체한다
  - 예: 곱셈/나눗셈 대신 **시프트 연산** 사용
- **낮은 정밀도 사용**: 단정밀도, 16비트, 허용되면 8비트까지 사용한다
- **하드웨어 내장 특수 함수** 활용: `rsqrtf()` 같은 하드웨어 제공 함수를 사용한다
  - 수학 라이브러리를 활용하면 자동으로 이러한 최적화가 적용되는 경우가 많다
- **분기문 감소**: 가능하면 조건부 실행(predicated execution)을 사용한다
- **원자 연산(atomic operation) 회피**: 원자 연산의 오버헤드는 크게 줄었지만, 여전히 피하는 것이 좋다
- **텐서 연산과 텐서 코어 활용**: 라이브러리를 사용하면 가능한 경우 자동으로 텐서 코어를 활용한다

### 최신 CUDA 기능

워프 수준 연산으로 **shuffle, vote, ballot** 등이 있다. 최신 GPU는 새로운 워프 수준 연산을 계속 추가하고 있다.

- 스레드 간 통신은 비용이 매우 크다
- 레지스터 파일은 스레드에 고유하므로, 공유 메모리 없이는 다른 스레드의 레지스터 내용에 접근할 방법이 없다
  - 레지스터 → 공유 메모리 → 레지스터로의 이동이 필요하며, 공유 메모리 뱅크 충돌을 유발할 수도 있다
- 따라서 **워프 내 데이터 이동**을 가능하게 하는 새로운 하드웨어 기능이 제공된다

최신 GPU의 주요 변화:

- **SIMT 실행에서 벗어나는 방향으로 진화**: Volta 아키텍처에서 도입된 **독립 스레드 스케줄링(Independent Thread Scheduling)** 등으로, 워프 내 스레드가 더 유연하게 독립 진행할 수 있다
- **공유 메모리를 여러 CUDA 블록이 공유 가능**: 이 강의에서는 공유 메모리가 하나의 CUDA 블록에서만 접근 가능하다고 가정하지만, 최신 GPU에서는 여러 CUDA 블록이 공유 메모리를 공유하도록 지정할 수 있다. 이는 해당 블록들이 같은 하드웨어 SM에 스케줄링될 때 의미가 있다.
- **통합 메모리(Unified Memory)**: 공유 메모리, L1 데이터 캐시, 텍스처 메모리의 통합
- **분리된 도착/대기 배리어(Split Arrive/Wait Barrier)** 의 하드웨어 가속: 동기화가 많은 경우에 비용이 크기 때문에 도입되었다

### CUDA 라이브러리

주요 CUDA 라이브러리:

| 라이브러리 | 용도 |
|-----------|------|
| **cuBLAS** | 선형대수 |
| **cuSPARSE** | 희소 행렬 |
| **cuTENSOR** | 텐서 기반 선형대수 |
| **Thrust** | 병렬 알고리즘 |
| **NCCL, NVSHMEM** | 멀티 GPU 통신 |
| **cuDNN** | 딥러닝 |

최신 라이브러리 문서를 참조하여 세부 사항을 확인한다.

### 기타 GPU 프로그래밍 플랫폼

이 강의는 CUDA에 초점을 맞추지만, 다른 프로그래밍 언어와 플랫폼도 존재한다:

- OpenCL, OneAPI/SYCL, OpenACC, HIP, Kokkos
- Python, Julia 같은 범용 언어에서도 GPU 프로그래밍이 가능하다

## 이 챕터에서 기억할 것

1. **GPU 최적화의 세 축** — 데이터 전송 시간, 메모리 접근 패턴, 연산 오버헤드를 모두 최적화해야 한다.
2. **피닝된 메모리와 비동기 전송** — `cudaMemcpyAsync`와 스트림으로 데이터 전송과 연산을 중첩·파이프라이닝할 수 있다.
3. **코얼레싱과 뱅크 충돌** — 전역 메모리는 코얼레싱·정렬을 보장하고, 공유 메모리는 뱅크 충돌을 줄여야 한다.
4. **낮은 정밀도와 특수 함수** — 가능한 한 낮은 비트 폭과 하드웨어 내장 함수를 사용하여 연산 비용을 줄인다.
5. **워프 수준 연산** — shuffle, vote 등으로 공유 메모리를 거치지 않고 워프 내 데이터를 직접 이동할 수 있다.
6. **최신 GPU 진화** — 독립 스레드 스케줄링, 블록 간 공유 메모리 공유, 통합 메모리 등으로 프로그래밍 모델이 유연해지고 있다.
